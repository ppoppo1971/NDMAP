# 최적화 체감도 및 렌더링 최적화 방안 보고서

**요청**: (1) 수정 전·후 최적화 체감도 평가, (2) 저사양 디바이스에서 부드러운 동작을 위한 렌더링 최적화 방안 보고. **프로그램 수정 없이 방안만 보고.**

---

## 1. 적용한 최적화의 체감도 평가

### 1.1 적용한 내용 요약

| 항목 | 내용 |
|------|------|
| A | 사진 모달 닫을 때 `img.src = ''` 로 data URL 참조 해제 |
| B | 내보내기 모달 닫을 때 `exportInfo = null` 로 blob 참조 해제 |
| C | 축척 표시 디바운스 100ms → 180ms |
| D | 저장 실패 시 alert 알림 (사진/텍스트/메모) |

### 1.2 체감도 결론

**전반적으로 “체감되는 최적화”는 거의 없다.**

- **프레임률·부드러움**: 변경 없음.  
  - A, B는 메모리 참조 해제만 하고, C는 축척 숫자 갱신 빈도만 줄였음.  
  - 스크롤/드래그/줌 시 프레임 드랍이 줄었다고 느끼기 어려움.
- **반응 속도**: 체감 차이 거의 없음.  
  - C로 인해 드래그 중 축척 숫자가 약간 덜 자주 바뀔 뿐, 지도 자체 반응은 동일.
- **체감이 있는 부분**  
  - **D (저장 실패 알림)**: 저장이 실패했을 때만 알림이 뜨므로, “기능 개선”에 가깝고, “속도/부드러움”과는 무관.  
  - **A·B**: 사진·내보내기를 많이 반복하는 장시간 사용 시, 메모리가 덜 쌓여서 **그런 경우에 한해** 멈춤·느려짐 예방에 도움될 수 있음. 일상적인 짧은 사용에서는 체감 거의 없음.

### 1.3 정리

- **수정 전·후로 “빨라졌다/부드러워졌다”는 체감은 기대하기 어렵다.**  
- 적용한 것은 **메모리 관리·저장 신뢰성** 쪽 개선에 가깝고, **렌더링·연산 부하**를 직접 줄이는 변경은 아니다.  
- 저사양에서 **눈에 띄게 부드럽게** 만들려면 **렌더링·그리기 부하**를 줄이는 별도 작업이 필요하다.

---

## 2. 저사양 디바이스용 렌더링 최적화 방안

아래는 **프로그램을 수정하지 않고** 방안만 정리한 것이다. 구현 시에는 저장/내보내기 무결성을 해치지 않는 범위에서 적용해야 한다.

---

### 2.1 Google Maps Data 레이어 (DXF 도면선)

**현상**  
- `map.data.addGeoJson(geoJson)` 로 모든 DXF feature를 한 번에 올림.  
- `map.data.setStyle(function (feature) { ... })` 는 **지도가 그릴 때마다** feature 수만큼 호출됨.  
- DXF가 크면 feature 수가 수백~수천 개가 되어, 저사양에서 줌/드래그 시 **프레임 드랍**이 날 수 있음.

**방안 (구현 난이도·효과)**  
- **뷰포트 컬링**: 현재 화면에 보이는 feature만 스타일 적용/표시하도록 제한.  
  - Google Maps Data 레이어는 “feature별 표시 여부” API가 제한적이어서, **GeoJSON을 뷰포트에 맞춰 잘라서 다시 addGeoJson**하는 방식이 필요할 수 있음.  
  - 줌/패닝 시 bounds 기준으로 GeoJSON을 필터링해 **일정 개수 이하**로 유지.  
  - **효과**: 높음 (feature 수가 많을 때). **난이도**: 중상.  
- **스타일 단순화**:  
  - strokeWeight, fillOpacity 등 변경 최소화.  
  - **효과**: 중간. **난이도**: 하.  
- **대형 DXF 분할**:  
  - 도면을 타일/구역 단위로 나누고, 보이는 구역만 Data 레이어에 로드.  
  - **효과**: 높음. **난이도**: 상.

---

### 2.2 텍스트 오버레이 (TextOnlyOverlay)

**현상**  
- `OverlayView` 의 `draw()` 가 **지도 이동·줌 시마다** 호출됨.  
- 매번 `this.div.innerHTML = ''` 후 텍스트 개수만큼 `span` 을 새로 만들고, `fromLatLngToDivPixel` 로 위치 계산.  
- 텍스트가 많으면 DOM 생성·배치 비용이 반복되어 저사양에서 **드래그/줌 시 끊김** 요인.

**방안**  
- **draw() 디바운스/쓰로틀**:  
  - `bounds_changed` / `idle` 에서만 `draw()` 를 호출하거나, **requestAnimationFrame** 으로 한 프레임에 한 번만 갱신.  
  - **효과**: 중. **난이도**: 하.  
- **뷰포트 밖 텍스트 생략**:  
  - `fromLatLngToDivPixel` 결과가 화면 밖이면 해당 `span` 은 만들지 않거나 숨김.  
  - **효과**: 텍스트 많을 때 중~높음. **난이도**: 중.  
- **DOM 재사용**:  
  - 매번 `innerHTML = ''` 대신 기존 `span` 을 재사용하고 위치·텍스트만 갱신.  
  - **효과**: 중. **난이도**: 중.

---

### 2.3 사진 마커 (Google Maps Marker)

**현상**  
- 사진 개수만큼 `google.maps.Marker` 생성.  
- 마커 하나당 DOM 노드와 이벤트가 있으므로, **수십~수백 개**가 되면 저사양에서 목록 스크롤·지도 이동 시 부담.

**방안**  
- **마커 클러스터링**:  
  - 줌 레벨이 작을 때는 근처 마커를 하나의 클러스터 아이콘으로 묶고, 확대 시 개별 마커로 펼침.  
  - **효과**: 마커 많을 때 높음. **난이도**: 중 (클러스터 라이브러리 도입 또는 직접 구현).  
- **뷰포트 내 마커만 표시**:  
  - 현재 bounds 안에 있는 사진만 마커로 그리기. 지도 이동 시마다 목록 갱신.  
  - **효과**: 높음. **난이도**: 중.  
- **캔버스 기반 일괄 그리기**:  
  - 마커를 DOM 대신 하나의 Canvas/OverlayView 로 그리기.  
  - **효과**: 매우 높음. **난이도**: 상 (클릭 영역·툴팁 등 처리 필요).

---

### 2.4 CSS·레이아웃

**현상**  
- 지도 영역이 크고, 모달·슬라이드 메뉴·버튼 등이 있어 **reflow/repaint** 가 자주 일어나면 저사양에서 끊김에 기여할 수 있음.

**방안**  
- **지도 컨테이너 격리**:  
  - `#map` 또는 그 부모에 `contain: layout paint` (또는 `content-visibility`) 적용해 지도 영역 reflow가 다른 UI로 전파되지 않도록 함.  
  - **효과**: 중. **난이도**: 하.  
- **고정 UI 레이어**:  
  - 햄버거, 줌 버튼, 축척 등 고정 요소에 `will-change: transform` 또는 `transform: translateZ(0)` 로 별도 레이어 분리 (과다 사용은 오히려 메모리 부담).  
  - **효과**: 소~중. **난이도**: 하.  
- **모달 표시 방식**:  
  - `display: none/block` 대신 `opacity` + `pointer-events` 또는 `visibility` 로 전환하면 reflow를 줄일 수 있음.  
  - **효과**: 소. **난이도**: 하.

---

### 2.5 DXF 로드 시 (첫 화면)

**현상**  
- `parser.parseSync(text)` 와 `extractConstantWidths` 가 **메인 스레드**에서 실행되면, 대형 DXF 로드 시 **첫 화면이 잠깐 멈춘 것처럼** 보일 수 있음.

**방안**  
- **Web Worker 에서 파싱**:  
  - DXF 텍스트만 Worker 로 넘기고, 파싱 + constantWidth 추출까지 Worker 에서 수행.  
  - 완료 후 메인 스레드에서는 `addGeoJson` / `applyDxfToMap` 만 수행.  
  - **효과**: 첫 로드 체감 개선에 유효. **난이도**: 중 (Worker ↔ 메인 간 데이터 직렬화·경계 설계 필요).  
- **로딩 표시**:  
  - 이미 “처리 중...” 표시가 있으므로, 파싱을 비동기로 나누면 “언제 끝날지” 체감은 더 나아질 수 있음.

---

### 2.6 기타

- **터치 이벤트**:  
  - `passive: true` 는 이미 사용 중이므로, 스크롤/제스처 시 불필요한 메인 스레드 블로킹은 이미 줄어든 상태.  
- **축척 표시**:  
  - 180ms 디바운스 적용으로, 드래그 중 JS 실행 횟수는 이미 줄어듦.  
  - 더 줄이려면 `idle` 에서만 갱신하는 방식으로 바꿀 수 있음 (숫자 갱신이 “드래그가 끝난 뒤”로만 됨).  
- **이미지 압축**:  
  - `compressImage` 는 사진 추가 시에만 실행되며, 지도 렌더링과는 별개.  
  - 저사양에서는 “한 번에 한 장” 처리 유지가 나을 수 있음 (동시 압축 수 제한 등은 선택 사항).

---

### 2.7 축척 기준 객체 표시 전환 (200m 기준)

**아이디어**  
- **축척(화면 가로 폭) 200m 이상**일 때: 블록(INSERT), 문자(TEXT/MTEXT) 등 세부 객체는 그리지 않음.  
- **200m 미만**일 때: 모든 객체(선·원·블록·문자 등) 표시.  
- 줌아웃 시에는 도면 골격(선·호 등)만 보이고, 줌인하면 문자·블록까지 보이게 해서 **그리기 부하를 축척에 따라 나누는** 방식.

**장점**  
- 구현이 **뷰포트 컬링**보다 단순할 수 있음: GeoJSON은 한 번만 올려두고, **스타일 콜백에서 “현재 축척”과 “feature 종류”만 보고 표시 여부를 결정**하면 됨.  
- 축척은 이미 `updateScaleDisplay`에서 계산하고 있으므로, 같은 식으로 “현재 가로 폭(m)”을 **캐시**해 두고, `map.data.setStyle(function (feature) { ... })` 안에서 이 캐시를 참조해, 예를 들어 `entityType`이 `TEXT` / `MTEXT` / `INSERT` 이고 축척 ≥ 200m 이면 `visible: false` (또는 투명)를 반환하면 됨.  
- **효과**: 문자·블록이 많은 DXF에서 줌아웃 시 그리기 수가 줄어들어, 저사양에서 드래그/줌 시 끊김 완화에 도움.  
- **저장/내보내기**: 표시만 바꾸는 것이므로 데이터·저장·내보내기 로직과 무관.

**구현 시 유의점**  
- GeoJSON feature에 **어떤 DXF 엔티티 타입에서 왔는지**를 알 수 있는 **속성 추가**가 필요함.  
  - 예: `entityType: 'LINE' | 'LWPOLYLINE' | 'CIRCLE' | 'ARC' | 'TEXT' | 'MTEXT' | 'INSERT' | 'POINT' | 'SPLINE'`  
  - INSERT는 “블록 전개” 시 나오는 선/폴리곤에도 `entityType: 'INSERT'` 또는 `fromBlock: true` 같은 표시를 해 두어야, 200m 이상에서 블록 전개 내용까지 숨길 수 있음.  
- **축척 값**은 `bounds_changed` / `idle` 시점에 한 번 계산해 **전역/캐시 변수**에 넣고, `setStyle` 콜백에서는 이 캐시만 읽도록 하면, getBounds()를 feature 수만큼 호출하는 일을 막을 수 있음.  
- 기준값(200m)은 상수로 두거나, 나중에 설정으로 바꿀 수 있게 해도 됨.

**정리**  
- **맞는 방법이다.**  
- “200m 이상에서는 블록·문자를 제외한 객체만, 200m 미만에서는 전부” 표시하는 방식은, **렌더링 부하를 축척에 따라 나누는** 하나의 실용적인 방안으로 보고서에 포함할 수 있다.  
- 우선순위는 Data 레이어 스타일만 바꾸면 되므로 **2.1(Data 레이어)과 함께 적용**하거나, **2.1 대안으로 먼저 적용**해 볼 만함. 효과는 **문자/블록 비중이 큰 도면**일수록 높다.

저사양에서 **부드러움**을 목표로 할 때, 효과와 구현 부담을 고려한 순서는 대략 다음과 같다.

## 3. 우선순위 제안 (렌더링 기준)

| 순위 | 대상 | 방안 요약 | 기대 효과 | 난이도 |
|------|------|-----------|-----------|--------|
| 1 | Data 레이어 (DXF) | **축척 200m 기준 표시 전환**: 200m 이상일 때 블록·문자(TEXT/MTEXT/INSERT) 미표시, 200m 미만일 때 전부 표시 | 높음(문자·블록 많은 도면) | 중 |
| 2 | Data 레이어 (DXF) | 뷰포트 기준 feature 수 제한 또는 컬링 | 매우 높음 | 중상 |
| 3 | 텍스트 오버레이 | draw 디바운스/RAF + 뷰포트 밖 생략 | 높음 | 하~중 |
| 4 | 사진 마커 | 클러스터링 또는 뷰포트 내만 표시 | 높음 | 중 |
| 5 | CSS | 지도 컨테이너 contain / 레이어 분리 | 중 | 하 |
| 6 | DXF 로드 | Web Worker 파싱 | 첫 로드 체감 개선 | 중 |

**참고**: 200m 축척 표시 전환(1번)은 GeoJSON은 그대로 두고 스타일만 바꾸면 되어, 2번(뷰포트 컬링)보다 구현이 단순하다. 먼저 1번을 적용한 뒤, 필요 시 2번을 추가하는 순서를 권장할 수 있다.

---

## 4. 요약

- **이번에 적용한 최적화(A~D)는 “체감되는 속도/부드러움” 개선은 거의 없고**, 메모리·저장 신뢰성 개선에 가깝다.  
- 저사양에서 **실제로 부드럽게** 만들려면 **렌더링·그리기 부하**를 줄이는 작업이 필요하며, 그중에서도 **축척 200m 기준으로 블록·문자만 숨기는 표시 전환**, **Data 레이어 feature 수 제한/컬링**, **텍스트 오버레이 draw 최소화**, **마커 수 제한/클러스터링**이 효과가 크다.  
- **축척 200m 이상일 때 블록·문자 미표시, 200m 미만일 때 전부 표시**하는 방식은 구현이 상대적으로 단순하고, 저장/내보내기와 무관하므로 우선 검토할 만한 방안이다.  
- 위 방안은 모두 **현재 프로그램을 수정하지 않고** 제안만 한 것이며, 적용 시에는 **저장/내보내기 무결성**을 해치지 않도록 설계해야 한다.
