다음 네 가지 최적화에 대해 **구현 내용**, **장단점**, **권장 수준**을 정리합니다.

## 1. bindContextMenuCloseOnMap()를 ensureMap()에서 1회만 호출하도록 이동

### 1.1 현재 동작

- **호출 위치**: `initMap()` (app.js 73행). Google Maps API 로드 후 콜백으로 실행되는 진입점.
- **지도 생성 시점**: 지도(`map`)는 `initMap()`이 아니라 **뷰어가 처음 표시될 때** `ensureMap()` 안에서 생성됨 (app.js 112행 `map = new google.maps.Map(...)`).
- **bindContextMenuCloseOnMap() 내부** (app.js 1468~1483):
  - `if (!map || !contextMenuEl) return;` → **map이 null이면 아무 리스너도 붙이지 않고 return**.
  - map이 있을 때만: `map.addListener('dragstart', ...)`, `map.addListener('zoom_changed', ...)`, `document.addEventListener('touchstart', ...)`, `document.addEventListener('mousedown', ...)` 4개 등록.

**결과**: `initMap()` 시점에는 `map === null`이므로 `bindContextMenuCloseOnMap()`은 **항상 곧바로 return**하고, document에 대한 "지도 밖 클릭 시 컨텍스트 메뉴 닫기" 리스너가 **한 번도 등록되지 않음**.  
→ 롱프레스로 연 컨텍스트 메뉴가 **지도 밖 터치/클릭으로는 닫히지 않을 수 있음**. (지도 드래그/줌 시에는 `dragstart`/`zoom_changed`로 닫힘.)

### 1.2 수정 내용 (제안)

- **initMap()에서 `bindContextMenuCloseOnMap();` 한 줄 제거.**
- **ensureMap()** 안에서, 지도를 생성한 직후 다른 map 바인딩과 함께 **1회만** 호출:
  - `map` 생성 직후 블록(151~158행 부근)에 `bindContextMenuCloseOnMap();` 추가.
  - 이미 `map`이 있을 때만 실행되는 상단 블록(84~92행)에도, `mapBindingsDone`이 false일 때 `bindContextMenuCloseOnMap()`를 함께 호출.

이렇게 하면 **지도가 실제로 생성된 뒤에만** 호출되므로, document의 touchstart/mousedown 리스너가 1회 등록되고 "지도 밖 클릭 시 메뉴 닫기"가 의도대로 동작함.

### 1.3 장점

- **의도한 UX 복구**: 지도 밖 터치/클릭 시 컨텍스트 메뉴가 닫혀 사용자 기대에 부합함.
- **중복 호출 없음**: `ensureMap()`은 지도 생성 시 1회, 이후에는 `mapBindingsDone`으로 바인딩을 1회만 수행하므로, `bindContextMenuCloseOnMap()`도 1회만 실행됨.
- **리스너 누수 없음**: document에 리스너를 한 번만 붙이므로, 나중에 "뷰어 해제" 플로우를 도입할 때만 그 시점에 제거를 고려하면 됨.
- **기존 동작 유지**: 사진/텍스트/저장/삭제 등 다른 기능에는 영향 없음.

### 1.4 단점

- 없음. 단순히 호출 시점을 "지도 생성 후"로 옮기는 것뿐이며, 기존에 동작하지 않던 부분이 동작하게 되는 수준임.

### 1.5 정리

| 구분 | 내용 |
|------|------|
| 권장도 | **강하게 권장** (버그 수정에 가까움) |
| 위험도 | 낮음 (호출 위치만 변경, 로직 변경 없음) |
| 공수 | 매우 작음 (한 줄 제거 + 두 위치에 한 줄 추가) |

## 2. TextOnlyOverlay.draw() throttle 또는 "bounds/zoom 변경 시에만" 재그리기

### 2.1 현재 동작

- **TextOnlyOverlay**: 사용자가 입력한 텍스트(메모)를 지도 위에 표시하는 오버레이. `google.maps.OverlayView` 상속.
- **draw()** (app.js 1426~1455): Maps API가 **pan/zoom 등 뷰 변경 시마다** 호출할 수 있음.
  - 매번 `this.div.innerHTML = ''` 로 자식 제거 후,
  - `textsArr`를 순회하며 각 텍스트에 대해 `span` 생성, `fromLatLngToDivPixel`로 픽셀 좌표 계산, `span`에 스타일·클릭 리스너 부착 후 `appendChild`.

**문제점**:

- pan/zoom을 연속으로 하면 **draw()가 짧은 시간에 여러 번** 호출될 수 있음.
- 호출마다 **전체 span 재생성 + 클릭 리스너 N개 재등록** → DOM 조작·이벤트 등록 비용이 반복됨.
- 텍스트 개수가 많을수록, 그리고 저사양 기기에서 배터리·메모리 부담이 커질 수 있음.

### 2.2 수정 방안 A: throttle (시간 기준)

- **내용**: draw() 실행을 **일정 간격(예: 100~150ms)으로만** 허용. 그 사이에 추가 draw 요청이 오면 무시하거나, 마지막 요청 시점을 기억해 한 번만 나중에 실행.
- **구현 예**: `drawScheduled` 플래그 + `setTimeout` 또는 `requestAnimationFrame`으로 "한 프레임에 한 번" 또는 "마지막 호출로부터 100ms 후 한 번" 실행.

**장점**: 구현이 비교적 단순; 빈번한 draw 호출을 확실히 줄일 수 있음.  
**단점**: throttle 구간 동안 뷰가 바뀌면 텍스트 위치가 잠깐 어긋나 보일 수 있음(간격이 짧으면 체감 거의 없음).

### 2.3 수정 방안 B: "bounds/zoom 변경 시에만" 재그리기

- **내용**: **이전 bounds와 zoom을 저장**해 두고, draw()가 호출될 때 **현재 bounds/zoom과 비교**해 실제로 바뀐 경우에만 `innerHTML` 비우고 span 재생성. 바뀌지 않았으면 early return.
- **구현 예**: 오버레이 인스턴스에 `_lastBounds`, `_lastZoom` 등을 두고, `getProjection()`으로 현재 bounds(또는 center+zoom)를 구한 뒤 비교.

**장점**: 불필요한 재그리기를 원천적으로 줄임; 화면이 안 바뀌면 DOM을 건드리지 않음.  
**단점**: bounds 비교 로직 추가 필요; Maps API가 draw()를 어떤 타이밍에 호출하는지에 따라 "한 번도 그리지 않음" 같은 엣지 케이스를 확인해야 함.

### 2.4 장점 (공통)

- **CPU/배터리**: pan/zoom 시 불필요한 DOM·리스너 생성 감소 → 메인 스레드 부하와 배터리 소모 완화.
- **메모리**: 짧은 시간에 많은 span/리스너가 반복 생성되는 상황 완화.
- **기능 유지**: 텍스트 표시·클릭 동작은 그대로 두고, **그리는 빈도만** 조절.

### 2.5 단점 / 주의

- **throttle**: 간격을 너무 길게 잡으면 드래그 중에 텍스트가 잠시 뒤쳐져 보일 수 있음 → 100~150ms 수준 권장.
- **bounds/zoom 비교**: 초기 draw()에서 한 번은 반드시 그리도록 하고, "변경 시에만"은 그 다음 호출부터 적용해야 함.
- **동작 검증**: 다양한 zoom/pan 패턴에서 텍스트가 빠짐없이 올바른 위치에 그려지는지 확인 필요.

### 2.6 정리

| 구분 | 내용 |
|------|------|
| 권장도 | **권장** (배터리·메모리·체감 성능 개선) |
| 위험도 | 낮음 (로직을 신중히 넣으면 기존 기능 유지 가능) |
| 공수 | throttle이 더 작음; bounds/zoom 비교는 중간 |
| 선택 | throttle이 구현·검증이 단순해 우선 적용하기 좋음. 필요 시 이후에 bounds/zoom 비교를 추가해 이중으로 제한할 수 있음. |

## 3. DXF 원문 / IMAGE ref 파싱을 로드 플로우에서 1회만 하도록 정리

### 3.1 현재 동작

- **로드 경로가 두 가지**:
  - **loadDxfFile(file)**: 파일 하나 선택 → `file.text()` 로 원문 `text` 획득 → `parser.parseSync(text)` → **extractConstantWidths(dxfData, text)** → **extractDxfImageRefs(text)** → 이후 applyDxfToMap, drawDxfImageMarkers 등.
  - **loadDxfFromFolder(files)**: 폴더에서 DXF 파일 하나 골라 동일하게 `dxfFile.text()` → `parser.parseSync(text)` → **extractConstantWidths(dxfData, text)** → **extractDxfImageRefs(text)** → 동일 후처리.

- **파싱/추출 함수**:
  - **extractConstantWidths(dxfData, dxfText)**: DXF **원문 문자열**을 `split(/\r?\n/)`로 줄 단위 순회하며 LWPOLYLINE/POLYLINE의 constantWidth(그룹코드 43) 등을 추출해 `dxfData` 엔티티에 보강. **원문 전체를 한 번 순회.**
  - **extractDxfImageRefs(dxfText)**: DXF **원문 문자열**을 다시 `split(/\r?\n/)`로 순회하며 OBJECTS의 IMAGEDEF, ENTITIES의 IMAGE를 파싱. **원문 전체를 두 번(OBJECTS, ENTITIES) 순회.**

**같은 DXF 로드 1회당**:
- `text`(원문)는 `file.text()` 또는 `dxfFile.text()`로 **1번** 읽음.
- `extractConstantWidths(dxfData, text)`에서 원문 **1회 순회**.
- `extractDxfImageRefs(text)`에서 원문 **다시 1~2회 순회**.
- 즉, **한 번의 로드** 안에서는 원문이 **2~3회** 순회됨. (같은 파일을 여러 번 연다면 그때마다 반복.)

### 3.2 수정 방향: "로드 플로우에서 1회만"

- **의미**: 한 번의 DXF 로드(loadDxfFile 또는 loadDxfFromFolder 한 번 호출) 안에서, **원문 문자열을 한 번만 순회**하거나, 순회 결과를 재사용하도록 정리.
- **방안 1 – 하나의 함수로 통합**:  
  `parseDxfAndExtras(text)` 같은 함수를 두고,  
  - `parser.parseSync(text)` → `dxfData`  
  - 같은 `text`로 constantWidth 추출 + IMAGE ref 추출을 **한 루프 또는 연속된 한 번의 순회**로 처리해 `{ dxfData, constantWidthMap, imageRefs }` 형태로 반환.  
  - loadDxfFile / loadDxfFromFolder는 이 함수를 **한 번만** 호출하고, 반환값으로 applyDxfToMap, dxfImageRefs 설정 등을 수행.
- **방안 2 – 원문 캐시 + 기존 함수 유지**:  
  현재처럼 `extractConstantWidths(dxfData, text)`, `extractDxfImageRefs(text)`를 따로 두되, **로드 함수 안에서는 `text`를 한 번만 읽고** 두 함수에 같은 `text`를 넘기기만 함.  
  - "1회만"의 의미를 **"같은 원문을 두 번 읽지 않는다"**로 제한하면 이미 만족함.  
  - "원문 순회 자체를 1회로 줄인다"까지 하려면 방안 1처럼 **한 번의 순회에서 constantWidth와 IMAGE ref를 동시에 추출**하는 로직으로 합치는 작업이 필요함.

### 3.3 장점

- **방안 2만 적용 시**: 코드 변경 최소. `file.text()` 결과를 한 번만 쓰고 두 추출 함수에 넘기는 것은 이미 대부분 그렇게 되어 있음. **실제 이득은 "중복 순회 제거"**이려면 방안 1 필요.
- **방안 1 (한 번의 순회로 통합)**:
  - 대용량 DXF에서 **원문 순회 횟수 감소** → 파싱 구간 CPU 시간 단축.
  - 로드 로직이 **한 곳(parseDxfAndExtras)**으로 모여 유지보수·테스트가 쉬워짐.
- **기능 유지**: DXF 표시, IMAGE ref(파란원), constantWidth 적용 결과는 동일하게 유지 가능.

### 3.4 단점

- **방안 1**: extractConstantWidths와 extractDxfImageRefs의 **파싱 로직을 하나로 합치거나** 호출 순서/데이터 구조를 바꿔야 함 → 리팩터 비용과 회귀 테스트 필요.
- **방안 2**: 원문을 두 번 읽지 않도록만 하면, 이미 1회 읽기로 두 번 넘기고 있어서 **추가 이득이 크지 않을 수 있음**. 진짜 이득은 **순회 횟수 감소**(방안 1).
- **대용량이 아닌 DXF**에서는 체감 성능 차이가 작을 수 있음.

### 3.5 정리

| 구분 | 내용 |
|------|------|
| 권장도 | **선택** (대용량 DXF·코드 정리 시 유리) |
| 위험도 | 방안 1은 중간 (파싱 로직 통합 시 버그 가능성); 방안 2는 낮음 |
| 공수 | 방안 2는 작음; 방안 1은 중간~큼 |
| 권장 순서 | 먼저 **로드 함수만 정리**(같은 `text`를 한 번만 읽고 두 추출 함수에 전달)로 통일감을 주고, 필요 시 **한 번의 원문 순회로 두 추출을 통합**하는 방안 1을 검토. |

## 4. 저장 실패 시 사용자 안내·재시도 또는 임시 보관 후 재시도

### 4.1 현재 동작

- **저장 API** (local-storage.js):
  - `saveProject`, `savePhoto`, `updatePhotoMemo` 등은 IndexedDB 트랜잭션을 사용하며, **tx.onerror** 시 `reject(tx.error)`로 Promise를 reject함.
- **호출 측** (app.js):
  - **savePhoto** (사진 추가): `.catch()`에서 `console.error` + **alert('사진 저장에 실패했습니다. 다시 시도해 주세요.')** (1714~1716행).
  - **saveProject** (텍스트 추가/수정/삭제): `.catch()`에서 `console.error` + **alert('텍스트 저장에 실패했습니다. 다시 시도해 주세요.')** 등 (1737~1742, 1887~1892, 1904~1909 등).
  - **updatePhotoMemo** (메모 저장): `.catch()`에서 **alert('메모 저장에 실패했습니다. 다시 시도해 주세요.')** (1832~1834).

**정리**:  
저장 실패 시 **사용자에게 alert로 안내는 것은 이미 구현되어 있음**.  
부족한 부분은 (1) **자동 재시도**, (2) **임시 보관 후 재시도**, (3) **실패 원인(디스크 부족, 권한 등)에 따른 구체 안내** 등이다.

### 4.2 개선 방안 A: 사용자 안내 강화

- **내용**:  
  - catch 시 **기존 alert 유지**하되, 메시지에 **에러 메시지 요약**을 포함 (예: `err && err.message ? err.message : '알 수 없음'`).  
  - 가능하면 **"다시 시도" 버튼**이 있는 작은 모달로 바꿔, 사용자가 한 번 더 저장을 시도할 수 있게 함.
- **장점**: 구현이 단순; 사용자가 "저장이 안 됐다"는 것을 명확히 알고, 수동으로 다시 시도할 수 있음.  
- **단점**: 자동 복구는 아님; 네트워크/디스크 일시 오류 시 사용자가 직접 다시 눌러야 함.

### 4.3 개선 방안 B: 제한적 자동 재시도

- **내용**:  
  - 저장 Promise가 reject되면 **최대 2~3회**, 짧은 지연(예: 1초) 후 **같은 payload로 다시 put** 시도.  
  - 모두 실패하면 기존처럼 alert(또는 모달)로 "저장에 실패했습니다. 다시 시도해 주세요." 안내.
- **장점**: 일시적인 오류(예: 브라우저가 잠시 IndexedDB를 사용 중인 경우)에서 **자동으로 성공할 가능성**이 있음.  
- **단점**:  
  - 영구적 실패(디스크 부족, 권한 없음)에서는 결국 alert까지 가므로, 사용자 경험은 "조금 늦게 실패 알림"이 됨.  
  - 재시도 로직을 saveProject/savePhoto/updatePhotoMemo 각각 또는 local-storage 레이어에 넣어야 함.

### 4.4 개선 방안 C: 임시 보관 후 재시도

- **내용**:  
  - 저장 실패 시 **메모리 또는 sessionStorage**에 "미저장 사진/텍스트/메모"를 **임시로 보관**.  
  - 주기적(예: 30초마다) 또는 "다음 포그라운드 진입 시"에 **저장 재시도**.  
  - 성공 시 임시 보관 제거; 여러 번 실패 시 사용자에게 "저장할 수 없습니다. 공간을 확보한 뒤 다시 시도해 주세요." 등으로 안내.
- **장점**:  
  - 앱을 닫지 않고 잠시 후(디스크/권한 복구 등) 자동으로 저장될 수 있어, **데이터 유실 가능성을 줄임**.  
- **단점**:  
  - 구현 복잡도가 큼 (임시 큐, 재시도 스케줄, 앱 생명주기 연동).  
  - sessionStorage/메모리는 탭 종료·크래시 시 사라지므로, "임시 보관"이 오래 지속되면 안 됨.  
  - **사진 blob**을 메모리에 오래 두면 메모리 부담이 커짐.

### 4.5 장점 (공통)

- **데이터 안전성**: 사용자가 "저장됐다"고 믿었는데 실제로는 실패한 상황을 줄이고, 실패 시 인지·재시도 가능성을 높임.
- **기존 기능 유지**: 성공 경로는 그대로 두고, **실패 시에만** 안내/재시도/임시 보관을 추가하는 형태로 설계 가능.

### 4.6 단점 / 주의

- **재시도**: 무한 재시도는 부하를 줄 수 있으므로 **횟수·간격 상한** 필요.
- **임시 보관**:  
  - 사진 blob을 메모리에 두는 것은 **용량·탭 종료 시 유실** 리스크가 있음.  
  - "임시 보관"은 **텍스트/메타** 위주로 하고, **사진은 "재시도 + 실패 시 안내"**까지로 제한하는 편이 안전함.
- **덮어쓰기 주의**: 재시도 시 **동일 키로 put**하므로, 그 사이에 사용자가 같은 항목을 수정했다면 Last-Write-Wins이 됨. 지금 구조와 동일한 동작이므로 기존 기능은 유지됨.

### 4.7 정리

| 구분 | 내용 |
|------|------|
| 권장도 | **안내 강화(A) 권장**; **재시도(B)**는 선택; **임시 보관(C)**은 요구·리소스에 따라 검토 |
| 위험도 | A는 낮음; B는 낮음(횟수 제한 시); C는 중간(복잡도·메모리) |
| 공수 | A는 작음; B는 중간; C는 큼 |
| 권장 순서 | 1) **에러 메시지 포함 + (선택) "다시 시도" 버튼**으로 안내 강화. 2) 필요 시 **2~3회 자동 재시도**를 local-storage 또는 호출 측에 추가. 3) 임시 보관은 **텍스트/메타만** 고려하고, 사진은 재시도와 명확한 실패 안내로 마무리하는 수준을 권장. |
